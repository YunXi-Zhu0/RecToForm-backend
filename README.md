# RecToForm-backend
一个AI Workflow项目, 基于 fastapi 和 llm 的发票信息提取与自动填表系统服务端，支持pdf、图片多种格式输入，批量识别提取发票信息并生成excel表格\
tips: 本项目为前后端分离架构, 本仓库仅为服务端; 前端展示和交互功能请移步 [RecToForm-frontend](https://github.com/YunXi-Zhu0/RecToForm-frontend)

# 二、服务端核心能力
面向二次开发者，提供了一套可嵌入业务系统的发票处理服务链路 \
核心能力包括：
- 任务接入
- 文件预处理
- 视觉模型抽取
- 结构化结果校验
- 模板填表导出
- 全流程审计追踪

## 1. API 化任务接入能力
- 基于 `FastAPI` 提供 HTTP 服务，默认支持跨域访问，便于前端页面、内部平台或第三方系统直接接入。
- 提供健康检查接口 `/health`，可用于部署探活、容器编排和网关监控。
- 提供异步任务创建接口，支持一次上传多个发票文件，适合批处理场景。
- 在任务创建阶段内置文件数量、文件大小、文件类型校验，并对重复上传文件做去重拦截。
- 提供任务状态查询接口，可轮询获取 `running / succeeded / partially_succeeded / failed` 等状态，以及阶段性进度信息。

## 2. 发票文件预处理能力
- 支持图片和 PDF 两类输入。
- 图片文件可直接进入视觉模型链路，避免不必要的二次转换。
- PDF 文件会按页渲染为图片，并保留页码顺序，保证后续识别结果与原始文档页面可追溯。
- 预处理阶段会生成 `manifest`，记录源文件、页码、输出图片路径等信息，便于排障和审计。
- 对 PDF 渲染图片提供可配置的增强处理，包括自动对比度、亮度/对比度增强、锐化，提升视觉模型识别稳定性。

## 3. 视觉模型调用与结构化抽取能力
- 后端已封装视觉语言模型调用流程，当前默认围绕 `qwen` 系列兼容能力设计，便于后续替换具体 provider。
- Prompt 构建显式区分 `system prompt` 与 `user prompt`：
  - `system prompt` 负责约束角色、字段范围、缺失值策略和 JSON 输出规则。
  - `user prompt` 负责描述当前任务、模板上下文和附加抽取要求。
- 模型调用默认要求返回 `json_object`，避免自然语言解释污染结果。
- 对模型原始输出提供 JSON 清洗与结构化解析逻辑，统一补全标准字段，降低上游模型波动对业务层的影响。
- 支持通过 `extra_instructions` 注入额外抽取要求，方便二开时按业务补充字段解释规则、票据识别侧重点或特殊格式约束。

## 4. 标准字段 Schema 能力
- 后端内置标准发票字段 schema，并通过独立服务统一加载和校验。
- 提供 `/standard-fields` 接口，可直接获取字段列表、schema 版本、缺失值占位规则，适合前端动态渲染字段选择器或表头编辑器。
- LLM 输出、模板映射、Excel 导出都围绕同一份标准字段 schema 运转，避免接口层、模型层、导出层字段定义不一致。
- 适合作为上层“字段中心”或“票据数据字典”模块的底层服务。

## 5. 模板元数据与字段映射能力
- 后端支持模板索引与模板配置装载，可列出当前可用模板并查询模板详情。
- 提供 `/templates` 与 `/templates/{template_id}` 接口，返回模板版本、映射版本、推荐字段、默认表头标签等元数据。
- 模板映射采用显式 `field_id -> sheet/cell` 配置，不依赖隐式猜测，便于审计、调试和版本管理。
- 模板层支持默认附加指令，可把某类模板对应的抽取偏好下沉到配置中，而不是散落在业务代码里。
- 适合扩展为“模板中心”“客户定制模板库”“多版本表单配置平台”。

## 6. Excel 生成与结果导出能力
- 支持两种结果输出模式：
  - 模板模式：按选定 Excel 模板和字段映射写入指定单元格。
  - 标准字段直出模式：直接按标准字段生成一份表格式 Excel，适合人工校对、数据复核或二次导入。
- 模板模式下支持动态同步表头名称，并清空未选字段对应的表头和数据单元格，便于实现“默认模板 + 可选字段”导出。
- 提供任务结果接口，可返回预览表头、预览数据行和 Excel 下载地址。
- 提供独立的标准表格导出接口 `/exports/standard-fields`，可将任意表头和行数据重新导出为 Excel，适合二开时承接在线编辑后的结果回写。

## 7. 批处理与任务编排能力
- 后端以任务为单位编排“上传 -> 预处理 -> LLM 抽取 -> JSON 校验 -> Excel 导出”全链路。
- 支持批量文件并发处理，并通过并发数配置控制吞吐与资源占用。
- 对批任务提供文件级成功/失败结果汇总，允许出现 `partially_succeeded`，适合真实业务中的混合质量输入场景。
- 文件级处理结果会回写到任务记录中，便于上层系统展示每个文件的识别状态、错误信息和产物路径。

## 8. 审计、排障与可运维能力
- 每个文件处理完成后都会持久化审计记录，包含：
  - 输入文件路径
  - 模板快照
  - 标准字段与导出字段
  - Prompt 上下文
  - 模型原始文本输出
  - 清洗后的结构化 JSON
  - 文档页图 manifest
  - 状态流转历史
  - 异常信息与 traceback
- 这意味着后端不仅能“给结果”，还能为二次开发提供可复盘的处理证据链。
- 服务启动后可启用输出目录清理调度，便于控制导出文件和中间产物的生命周期。

## 9. 当前适合承接的二开场景
- 为业务前端提供发票上传、轮询进度、结果预览、Excel 下载的一整套后端接口。
- 作为内部 RPA、财务共享平台、采购报销系统的票据抽取服务。
- 作为“模板管理 + 字段映射 + 结构化识别”底座，被更上层的工作流系统二次封装。
- 在保留现有 API 协议的前提下，替换视觉模型 provider、补充模板、扩展字段或接入对象存储。

## 10. 对二次开发者的扩展建议
- 如果要新增票据类型，优先扩展标准字段 schema、Prompt 规则和模板映射，不要把字段逻辑直接写进路由层。
- 如果要切换模型，建议沿用 `src/integrations/llm/` 的 provider 抽象，保持业务编排层稳定。
- 如果要做长期任务管理，建议将当前任务仓储、队列和导出注册能力接入 Redis/数据库做持久化增强。
- 如果要做前后端联动配置，推荐围绕 `/templates`、`/standard-fields`、`/tasks`、`/exports` 这几类接口构建页面能力。


# 三、快速开始
### 1. Docker 启动（推荐）
先复制环境变量文件，并将 `.env` 中的 Redis 地址改为容器服务名：

```bash
cp .env.example .env
```

```env
REDIS_URL=redis://redis:6379/0
```

然后启动：

```bash
docker compose up --build
```

验证：

```text
http://127.0.0.1:8080/health
```

补充说明 `.env` 环境变量的配置
- `LLM_PROVIDER`：指定视觉语言模型提供商，默认选择 `qwen_official`
- 前往[阿里云官网](https://help.aliyun.com/zh/model-studio/get-api-key?spm=a2c4g.11186623.0.0.53bf172azgVCaY)获取 `QWEN3_VL_PLUS_API_KEY`，并将其添加到 `.env` 文件中：


### 2. 原生后端启动
复制环境变量并配置(同上)
```bash
cp .env.example .env
```
同步依赖并激活环境变量
```bash
uv sync
source .venv/bin/activate
```

分别启动 Redis、worker、后端（Redis需要提前安装）：

```bash
redis-server
```

```bash
uv run rq worker invoice_tasks
```

```bash
uv run uvicorn src.api.app:app --host 127.0.0.1 --port 8080
```

验证：

```text
http://127.0.0.1:8080/health
```

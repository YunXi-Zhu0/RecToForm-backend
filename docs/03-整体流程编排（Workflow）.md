# 发票信息自助提取填表程序：整体流程编排（Workflow）

## 1. Workflow 目标

本流程用于串联以下核心链路：

`读取发票 -> OCR -> Markdown -> 提示词构造 -> LLM 分析 -> JSON 校验清洗 -> Excel 填表 -> 结果返回`

该流程应由 `src/services/workflow/` 负责主编排，API 层只负责触发和查询，不承担完整业务链路的内部细节。

## 2. 主流程阶段划分

### 阶段 1：接收任务

输入内容：

- 发票文件
- 模板 ID
- 默认字段集合
- 用户额外勾选的可选字段集合

处理动作：

- 校验请求参数
- 校验文件类型与大小
- 创建任务 ID
- 初始化任务状态

输出内容：

- 任务已创建
- 初始状态记录

### 阶段 2：发票文件预处理

处理动作：

- 识别文件格式
- 若有需要，执行图片或 PDF 的基础预处理
- 生成后续 OCR 所需输入

注意点：

- 该阶段只做必要预处理，不改变票面关键信息。
- 预处理目标是提升 OCR 稳定性，而不是重构内容。

### 阶段 3：OCR 调用

处理动作：

- 调用 PaddleOCR-VL-1.5 API
- 获取 OCR 原始响应
- 记录原始响应与必要日志

输出内容：

- OCR 原始数据
- OCR 识别状态

异常处理：

- 网络异常
- API 超时
- 认证失败
- 响应结构异常

### 阶段 4：Markdown 中间结果生成

处理动作：

- 从 OCR 响应中提取文本与表格信息
- 整理为适合 LLM 消费的 Markdown
- 尽量保留票面表格结构与字段上下文

输出内容：

- Markdown 文本
- 可选的 Markdown 文件

质量要求：

- 字段与票面尽量一一对应
- 不丢失金额、日期、号码、税额等关键内容
- 表格关系尽量保真

### 阶段 5：模板与字段解析

处理动作：

- 加载模板元数据
- 获取模板默认字段
- 合并用户额外勾选的可选字段
- 生成本次任务的目标字段集合

输出内容：

- 模板信息
- 最终待提取字段列表
- 字段到 Excel 的映射配置

### 阶段 6：提示词构造

处理动作：

- 构造系统提示词
- 构造用户提示词
- 将 Markdown、模板信息、默认字段和可选字段纳入提示词

输出内容：

- system prompt
- user prompt

要求：

- 明确模型角色
- 明确输出 JSON 格式
- 明确禁止猜测字段
- 明确缺失值处理规则

### 阶段 7：LLM 调用

处理动作：

- 调用 DeepSeek API
- 传入系统提示词和用户提示词
- 获取原始模型输出

输出内容：

- LLM 原始响应

异常处理：

- 网络异常
- 超时
- 非 JSON 输出
- 输出字段不完整

### 阶段 8：JSON 清洗与校验

处理动作：

- 解析 JSON
- 清理多余文本
- 过滤无关字段
- 标准化空值
- 校验必填字段、字段类型和格式

输出内容：

- 合法的结构化 JSON
- 校验失败时的错误信息

### 阶段 9：Excel 填表

处理动作：

- 读取对应模板
- 根据字段映射写入 Excel
- 保持模板格式尽量不被破坏
- 生成输出文件

输出内容：

- 最终 Excel 文件
- 文件路径、下载信息或文件对象

### 阶段 10：任务结果返回

处理动作：

- 更新任务最终状态
- 返回结构化结果摘要
- 返回下载信息或错误信息

输出内容：

- 成功响应或失败响应

## 3. 状态流转建议

建议任务状态至少包含：

- `created`
- `validating`
- `ocr_processing`
- `markdown_generated`
- `prompt_ready`
- `llm_processing`
- `json_validated`
- `excel_generating`
- `succeeded`
- `failed`

## 4. 异步编排建议

### 4.1 编排位置

- API 层使用 `src/api/services/` 处理异步请求接入。
- 主业务编排放在 `src/services/workflow/`。

### 4.2 异步要求

- 统一使用 `async` / `await`
- 涉及任务调度时使用 `asyncio`
- 避免在主事件循环中引入阻塞式 I/O

### 4.3 建议的任务拆分

- 上传与任务创建
- OCR 调用任务
- Markdown 转换任务
- LLM 分析任务
- JSON 清洗校验任务
- Excel 生成任务

## 5. 中间结果留存建议

为便于排查问题，建议保留以下中间结果：

- OCR 原始响应
- OCR Markdown
- 模型调用时的提示词
- LLM 原始响应
- 清洗前 JSON
- 清洗后 JSON
- Excel 输出路径

## 6. 失败处理与回滚思路

### 6.1 OCR 阶段失败

- 记录原始错误
- 更新任务状态为失败
- 返回 OCR 失败原因

### 6.2 LLM 阶段失败

- 保存原始输出
- 进入清洗失败或解析失败分支
- 返回字段提取失败原因

### 6.3 Excel 阶段失败

- 保留已生成的结构化 JSON
- 返回模板写入失败原因

### 6.4 通用失败原则

- 错误信息要可定位
- 中间结果尽量可追踪
- 不因单一异常吞掉整条错误链路

## 7. 模块协作关系

建议模块调用关系如下：

1. `src/api/routes/` 接收请求
2. `src/api/services/` 创建异步任务
3. `src/services/workflow/` 进行主流程编排
4. `src/integrations/ocr/` 调用 PaddleOCR API
5. `src/services/ocr/` 生成 Markdown
6. `src/services/template/` 解析模板与字段
7. `src/services/llm/` 构造提示词并处理模型输出
8. `src/integrations/llm/` 调用 DeepSeek API
9. `src/services/excel/` 写入 Excel
10. API 层返回任务结果

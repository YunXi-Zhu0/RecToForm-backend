# 发票信息自助提取填表程序：提示词规范

## 1. 目标

提示词规范用于约束 DeepSeek API 的输入方式与输出结果，确保模型能够基于 OCR Markdown 稳定产出结构化 JSON，而不是自由文本描述。

## 2. 提示词分层原则

LLM 调用时必须拆分为两部分：

- 系统提示词
- 用户提示词

不建议将所有信息混合为一个大段文本，否则会降低约束清晰度和后续维护性。

## 3. 系统提示词规范

### 3.1 作用

系统提示词用于定义模型角色、行为边界、输出格式、字段约束和缺失值处理规则。

### 3.2 必须包含的内容

- 角色定义：模型是发票字段提取助手
- 输入说明：输入内容来自 OCR 结果，可能包含表格和噪声
- 输出要求：严格输出 JSON
- 字段约束：仅输出要求字段，不输出无关字段
- 缺失值规则：找不到字段时返回空值或约定缺失标记
- 幻觉约束：禁止猜测、补造、臆断字段值
- 命名规范：字段命名必须与模板字段定义一致

### 3.3 推荐约束语义

建议系统提示词明确表达：

- 你只能根据提供的 OCR Markdown 提取字段。
- 若某字段无法确认，必须返回空值，不能猜测。
- 最终输出必须为合法 JSON，且不得包含额外解释文本。
- 输出字段名必须与给定字段列表完全一致。

## 4. 用户提示词规范

### 4.1 作用

用户提示词用于提供本次具体任务上下文，包括模板信息、目标字段和 OCR Markdown 内容。

### 4.2 必须包含的内容

- 当前选择的模板信息
- 默认字段列表
- 用户勾选的可选字段列表
- 最终目标字段列表
- OCR 输出的 Markdown 内容
- 期望 JSON 结构或字段示例

### 4.3 推荐结构

建议按以下顺序组织用户提示词：

1. 当前任务说明
2. 当前模板信息
3. 必须提取的字段列表
4. 用户附加字段列表
5. 输出 JSON 示例
6. OCR Markdown 正文

## 5. 字段规则

### 5.1 字段来源

字段来源由三部分组成：

- 模板默认字段
- 用户勾选的可选字段
- 系统内部保留的必要控制字段

### 5.2 字段输出原则

- 只输出最终目标字段集合
- 字段名必须稳定
- 字段值尽量保留原始票面表达
- 无法确认的字段统一输出空值或缺失标记

### 5.3 常见字段类型建议

- 文本类：购方名称、销方名称、发票号码
- 日期类：开票日期
- 金额类：金额、税额、价税合计
- 编号类：税号、校验码、机器编号

### 5.4 标准化建议

- 日期统一格式
- 金额统一为可解析字符串或数值字符串
- 去除无关前后空白
- 对 OCR 错别字符进行必要校正，但禁止主观猜测

## 6. 输出 JSON 规范

### 6.1 总体要求

- 必须输出合法 JSON
- 不允许输出 Markdown 代码块包裹
- 不允许输出解释性文字
- 不允许追加“说明”“备注”“分析过程”

### 6.2 建议格式

```json
{
  "发票号码": "",
  "开票日期": "",
  "购方名称": "",
  "销方名称": "",
  "金额": "",
  "税额": "",
  "价税合计": ""
}
```

### 6.3 缺失值策略

若字段缺失，建议统一使用以下之一：

- 空字符串 `""`
- `null`

具体选型应在系统提示词中固定，不应在不同请求中频繁变化。

## 7. Prompt 模板建议

### 7.1 系统提示词模板示例

```text
你是发票字段提取助手。你的任务是根据提供的 OCR Markdown 内容提取目标字段，并严格输出 JSON。

要求：
1. 只能根据输入内容提取字段，禁止猜测或补造不存在的信息。
2. 若字段无法确认，返回空值。
3. 输出字段名必须与给定字段列表完全一致。
4. 最终只输出合法 JSON，不要输出解释、分析过程或额外文本。
```

### 7.2 用户提示词模板示例

```text
当前模板：模板A
默认字段：["发票号码", "开票日期", "购方名称", "金额"]
可选字段：["税额", "价税合计"]
最终目标字段：["发票号码", "开票日期", "购方名称", "金额", "税额", "价税合计"]

请根据以下 OCR Markdown 提取字段，并输出 JSON：

{markdown_content}
```

## 8. 异常输出防御策略

### 8.1 常见异常

- 模型输出非 JSON
- 模型在 JSON 外输出解释文本
- 模型输出多余字段
- 模型遗漏必填字段
- 模型用猜测值填补缺失字段

### 8.2 防御手段

- 在系统提示词中反复强调“只输出 JSON”
- 在用户提示词中明确目标字段清单
- 在服务层增加 JSON 清洗与校验
- 对非法输出走重试、报错或兜底逻辑

## 9. 与 Workflow 的关系

提示词构造阶段属于主 workflow 中的核心一环，位于：

`OCR Markdown 生成 -> 模板字段解析 -> 提示词构造 -> LLM 调用`

因此提示词规范不能脱离模板配置、字段映射和 JSON 校验独立设计。

## 10. 后续可继续细化的方向

若后续需要，可以继续增加：

- 字段级提示词模板
- 不同发票类型的专用 prompt
- JSON Schema 驱动的自动提示词生成
- 模型失败重试 prompt 策略

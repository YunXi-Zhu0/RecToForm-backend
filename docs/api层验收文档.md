# API层验收文档

## 当前设计结论

截至本轮需求澄清完成，API 层已确认以下核心口径：

1. 前端存在两种业务模式：
   - 默认模板模式
   - 自定义表头模式
2. 自定义表头模式下，后端不做字段绑定，不把用户填写的表头与模板字段做匹配。
3. 自定义表头模式下，后端统一返回 `template/standard_fields.json` 中的全部标准字段与识别结果。
4. 用户在前端表格中自行编辑表头、删列、调列顺序、补改单元格内容后，再发起导出。
5. 多文件分析必须走异步任务制，不采用单请求同步阻塞完成。
6. 任务调度采用两层并发模型：
   - `RQ worker` 负责进程级并发
   - 单个任务内部使用 `asyncio` 协程并发调度 services
7. `RQ worker` 进程数与单任务协程并发数必须进入 `src/core/config.py` 与 `.env`，供部署与运维调整。

因此，API 层不再只是“上传并返回 Excel”，而是承担以下职责：

- 接收并校验上传文件
- 创建异步任务并投递到队列
- 提供任务状态查询接口
- 返回结构化预览数据
- 区分模板模式与自定义表头模式
- 接收前端编辑后的标准字段表格并导出 Excel

---

## 验收目标

本文档用于验收后续 `src/api/` 是否已经围绕以下主链路落地：

`前端选择模式 -> 上传多文件 -> API 创建任务 -> RQ worker 调度 workflow -> 前端轮询状态 -> 获取预览数据 -> 下载或导出 Excel`

---

## 验收范围

覆盖以下模块与能力：

- `src/api/app.py`
- `src/api/routes/`
- `src/api/schemas/`
- `src/api/services/`
- `src/core/config.py`
- `.env` / `.env.example`
- 任务调度所需的 RQ 与 Redis 配置接入

同时覆盖以下交互模式：

- 默认模板模式
- 自定义表头模式

## 一、接口模式验收

### 验收目标

API 层能够明确区分模板模式与自定义表头模式，并对两种模式返回不同但清晰的结果契约。

### 必须满足的验收项

1. 请求模型中存在显式 `mode` 字段。
2. `mode` 至少支持：
   - `template`
   - `standard_edit`
3. 模板模式下允许传入 `template_id`、`template_version`。
4. 自定义表头模式下不要求传入模板字段绑定信息。
5. 自定义表头模式返回的是标准字段全集结果，而不是模板裁剪后的字段子集。

### 验收方法

- 构造模板模式请求
- 构造自定义表头模式请求
- 校验 API 对两种模式的参数校验与响应结构

### 通过标准

- 模式边界清晰
- 不存在“自定义表头模式仍要求后端做字段绑定”的行为

---

## 二、模板与标准字段元数据接口验收

### 验收目标

前端能够通过 API 获取模板列表和标准字段全集，为两种模式初始化界面。

### 必须满足的验收项

1. 提供模板列表接口。
2. 提供单模板详情接口。
3. 提供标准字段列表接口。
4. 标准字段接口返回 `standard_fields.json` 中全部字段及版本号。
5. 模板接口只服务于模板模式，不反向约束标准字段全集。

### 建议接口

- `GET /api/v1/templates`
- `GET /api/v1/templates/{template_id}`
- `GET /api/v1/standard-fields`

### 通过标准

- 前端无需读取本地静态文件即可完成初始化
- 模板元数据与标准字段元数据来源清晰分离

---

## 三、任务创建与上传接口验收

### 验收目标

API 层能够接收多文件上传，创建异步任务，并立即返回可轮询的 `task_id`。

### 必须满足的验收项

1. 支持多文件上传。
2. 上传接口不直接同步执行完整识别流程。
3. 上传成功后立即返回 `task_id`、初始状态和文件数量。
4. 文件类型、空文件、大小、数量等异常具备明确校验。
5. 上传接口能够把任务投递到 RQ 队列。

### 建议接口

- `POST /api/v1/tasks`

### 建议响应内容

- `task_id`
- `status`
- `mode`
- `total_files`

### 通过标准

- 前端上传后进入轮询态
- API 不承担长时间阻塞式识别执行

---

## 四、任务状态查询接口验收

### 验收目标

前端能够轮询任务状态，并获得足够的进度信息驱动界面展示。

### 必须满足的验收项

1. 提供单任务状态查询接口。
2. 状态返回包含阶段信息。
3. 状态返回包含文件级进度统计。
4. 失败时返回明确错误信息。
5. 支持区分：
   - 全部成功
   - 部分成功
   - 全部失败

### 建议接口

- `GET /api/v1/tasks/{task_id}`

### 建议状态集合

- `queued`
- `running`
- `file_preprocessing`
- `llm_processing`
- `assembling_results`
- `excel_generating`
- `succeeded`
- `partially_succeeded`
- `failed`

### 建议统计字段

- `total_files`
- `processed_files`
- `succeeded_files`
- `failed_files`
- `progress_percent`

### 通过标准

- 前端无需猜测任务进度
- 任务失败可追踪

---

## 五、结果返回接口验收

### 验收目标

API 层能够根据不同模式返回可直接消费的结果数据，而不是只返回一个二进制文件。

### 必须满足的验收项

1. 模板模式下可返回预览数据和 Excel 下载信息。
2. 自定义表头模式下返回标准字段全集的多行结果。
3. 结果接口返回文件级失败摘要。
4. 前端可以直接使用结果接口数据渲染表格，而不必先解析 xlsx。

### 建议接口

- `GET /api/v1/tasks/{task_id}/result`
- `GET /api/v1/tasks/{task_id}/excel`

### 自定义表头模式下建议返回

- `standard_fields`
- `rows`
- `failed_items`
- `task_id`
- `status`

### 模板模式下建议返回

- `preview_headers`
- `preview_rows`
- `excel_download_url`
- `failed_items`

### 通过标准

- 自定义表头模式可直接进入前端在线编辑
- 模板模式可直接进入下载或预览

---

## 六、自定义表格导出接口验收

### 验收目标

当前端已完成表头和数据编辑后，API 层能够接收最终二维表并生成导出文件。

### 必须满足的验收项

1. 存在独立导出接口。
2. 导出接口接收前端最终编辑后的 `headers + rows`。
3. 导出接口不要求用户再提交字段绑定信息。
4. 导出结果为可下载 Excel。

### 建议接口

- `POST /api/v1/exports/standard-fields`

### 建议请求内容

- `headers`
- `rows`
- `filename`

### 通过标准

- 自定义表头模式完整闭环成立
- 后端职责保持为“导出最终用户编辑结果”

---

## 七、任务调度与并发验收

### 验收目标

任务调度采用 `RQ + asyncio` 两层并发模型，并且所有关键并发参数均可配置。

### 必须满足的验收项

1. API 层不直接执行长耗时任务，而是投递到 RQ。
2. worker 进程数可配置。
3. 单个任务内的异步并发数可配置。
4. 配置项统一由 `src/core/config.py` 加载，不在其他层散落读取环境变量。
5. `.env` 中存在部署可调参数说明。

### 建议配置项

- `REDIS_URL`
- `RQ_QUEUE_NAME`
- `RQ_JOB_TIMEOUT`
- `RQ_RESULT_TTL`
- `RQ_WORKER_PROCESSES`
- `WORKFLOW_ASYNC_CONCURRENCY`
- `MAX_UPLOAD_FILES`
- `MAX_FILE_SIZE_MB`

### 实现约束

1. `RQ worker` 负责进程级并发。
2. 单个任务内部用 `asyncio` + `Semaphore` 控制文件级并发。
3. 阻塞式文件处理与 Excel 写入需避免直接卡住事件循环。

### 通过标准

- 任务调度模型与运维配置边界清晰
- 并发参数可通过配置稳定调整

---

## 八、与 services 层协作边界验收

### 验收目标

API 层只负责接入、校验、调度与查询，不把业务编排直接混入路由层。

### 必须满足的验收项

1. 路由层不直接拼装大模型调用逻辑。
2. 路由层不直接处理模板映射与 Excel 写入细节。
3. 主业务流程仍由 `src/services/workflow/` 负责。
4. API 层通过 `src/api/services/` 对任务投递、状态查询、结果整理进行封装。

### 通过标准

- 路由层保持薄
- API 层与 services 层职责清晰

---

## 当前阶段的最终验收判断

后续 API 层若完成以下结果，即可视为本阶段验收通过：

1. 前端可查询模板列表和标准字段全集。
2. 前端可按两种模式发起多文件异步任务。
3. 后端可通过 RQ 投递任务，并通过配置控制进程数和协程并发数。
4. 前端可轮询任务状态并显示进度。
5. 模板模式可返回预览信息与下载信息。
6. 自定义表头模式可返回标准字段全集多行结果，供前端编辑。
7. 前端编辑完成后可再次提交二维表并导出 Excel。

达到以上条件后，API 层就不再处于概念设计状态，而是进入“可承接前端联调”的阶段。
